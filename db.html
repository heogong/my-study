01
핵심 개념 & 용어
Fundamentals
키(Key) 종류
  • 기본키(PK) — 행을 유일하게 식별, NOT NULL + UNIQUE
  • 외래키(FK) — 다른 테이블 PK 참조, 무결성 보장
  • 후보키 — PK가 될 수 있는 최소 속성 집합
  • 슈퍼키 — 유일성 만족(최소성 불필요)
  • 대리키(Surrogate) — 의미 없는 인조 식별자(auto_increment 등)
무결성 제약
  • 개체 무결성 — PK는 NULL 불가
  • 참조 무결성 — FK는 존재하는 PK값 또는 NULL
  • 도메인 무결성 — 허용된 값 범위(CHECK, type)
  • NULL 무결성 — 필수 컬럼 NOT NULL 강제
카디널리티 & 관계
  • 1:1 — 사용자 ↔ 사용자상세
  • 1:N — 팀 ↔ 팀원 (가장 흔함)
  • N:M — 학생 ↔ 수업 → 중간테이블 필요
  • 카디널리티 — 컬럼 내 유니크 값 수, 인덱스 효율에 영향
02
정규화 (Normalization)
Design
목적

이상(Anomaly) 제거 — 삽입 이상 · 갱신 이상 · 삭제 이상을 방지하고 데이터 중복을 최소화

1
1NF — 원자값(Atomic): 반복 그룹 제거, 각 셀에 단일 값만 허용
2
2NF — 완전 함수 종속: 복합 PK 사용 시, 모든 일반 속성이 PK 전체에 종속 (부분 종속 제거)
3
3NF — 이행 함수 종속 제거: A→B→C에서 A→C 형태의 이행 종속 분리
BC
BCNF — 결정자가 모두 후보키여야 함. 3NF보다 강한 조건, 이상이 여전히 존재하는 경우 적용
4
4NF / 5NF — 다치 종속·조인 종속 제거. 실무에서는 3NF~BCNF까지가 일반적
반정규화(Denormalization) — 성능을 위해 의도적으로 중복 허용. 조회 빈도 높고 조인이 비싼 경우 적용. 쓰기 성능 저하 & 데이터 불일치 위험을 감수해야 함.
03
인덱스 (Index)
Performance
B-Tree 인덱스
  • MySQL InnoDB 기본 구조 (B+ Tree)
  • 리프 노드에 실제 데이터 포인터 저장
  • 범위 검색(<, >, BETWEEN, LIKE 'abc%')에 유리
  • = 검색, ORDER BY, GROUP BY에도 효율적
Hash 인덱스
  • = 검색만 O(1) 지원
  • 범위 검색 불가, 정렬 불가
  • Memory 엔진, Redis 등에서 활용
클러스터드 vs 논클러스터드
  • Clustered — 데이터가 인덱스 순서로 물리 저장. 테이블당 1개만 (InnoDB PK)
  • Non-clustered — 별도 인덱스 구조 → 리프에서 PK로 재조회(Bookmark Lookup)
  • 커버링 인덱스 — 쿼리에 필요한 컬럼이 인덱스에 모두 포함 → 테이블 접근 0회
인덱스 사용 주의사항
  • Index Range Scan 유도 (IS NULL, !=, % 앞 LIKE 회피)
  • 복합 인덱스 — 선두 컬럼 = 조건 필수 (좌측 접두사 규칙)
  • 카디널리티 낮은 컬럼 (성별 등) 단독 인덱스 비효율
  • 인덱스 과다 → INSERT/UPDATE 성능 저하
04
트랜잭션 & ACID
Transaction
ACID 속성
  • Atomicity — 전부 성공 or 전부 실패 (ROLLBACK)
  • Consistency — 트랜잭션 전후 DB 무결성 유지
  • Isolation — 동시 트랜잭션 간 간섭 방지
  • Durability — 커밋된 데이터는 장애 후에도 보존 (WAL)
동시성 문제
  • Dirty Read — 커밋 안 된 데이터를 읽음
  • Non-Repeatable Read — 같은 행 재조회 시 값이 다름
  • Phantom Read — 범위 재조회 시 행 수가 다름
  • Lost Update — 두 트랜잭션이 동시에 같은 행 수정
격리 수준 (Isolation Level)
격리 수준 Dirty Read Non-Repeatable Phantom 성능
READ UNCOMMITTED가능가능가능최고
READ COMMITTED 기본(PG)방지가능가능높음
REPEATABLE READ 기본(MySQL)방지방지가능보통
SERIALIZABLE방지방지방지최저
InnoDB REPEATABLE READ는 MVCC(다중 버전 동시성 제어)로 Phantom Read도 대부분 방지. SELECT … FOR UPDATE 시에는 Next-key Lock 사용.
05
락(Lock) & 동시성 제어
Concurrency
락 종류
  • Shared Lock (S) — 읽기 락, 여러 트랜잭션 동시 획득 가능
  • Exclusive Lock (X) — 쓰기 락, 단독 점유
  • Intent Lock — 계층 락(테이블→행)에서 상위 레벨 의도 표시
  • Gap Lock — 범위 사이 삽입 방지 (Phantom 방지)
낙관적 vs 비관적 락
  • 낙관적(Optimistic) — 충돌 적다고 가정, 커밋 시 버전 비교 → 실패 시 RETRY
  • 비관적(Pessimistic) — 충돌 자주 발생, SELECT FOR UPDATE로 선점
  • 읽기 多 → 낙관적 / 쓰기 多·충돌 高 → 비관적
데드락(Deadlock)
  • 두 트랜잭션이 서로의 락을 대기 — 교착 상태
  • 예방 — 항상 동일 순서로 자원 획득
  • 탐지 & 해결 — DBMS가 자동 감지 후 한 쪽 ROLLBACK
  • 타임아웃 — innodb_lock_wait_timeout 설정
06
쿼리 튜닝
Tuning
EXPLAIN 읽기
  • type — ALL(풀스캔) > index > range > ref > eq_ref > const
  • key — 실제 사용된 인덱스
  • rows — 예상 조회 행 수 (낮을수록 ↑)
  • Extra — Using filesort / Using temporary 등장 시 주의
  • filtered — WHERE 조건에서 걸러지는 비율
자주 쓰는 튜닝 패턴
  • WHERE 컬럼에 함수 감싸지 않기 → 인덱스 무력화
  • SELECT * 지양 → 필요한 컬럼만 지정
  • N+1 문제 → JOIN 또는 Batch 조회로 해결
  • OFFSET 大 → 커서 기반 페이징으로 대체
  • 서브쿼리 → 인라인뷰 or JOIN으로 개선
  • UNION → UNION ALL (중복 제거 불필요 시)
-- 인덱스 무력화 예시 (나쁜 쿼리) SELECT * FROM orders WHERE YEAR(created_at) = 2024; -- 함수 → Full Scan -- 개선: 범위 조건으로 인덱스 활용 SELECT * FROM orders WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'; -- 커서 페이징 (OFFSET 대신) SELECT * FROM orders WHERE id > :last_id ORDER BY id LIMIT 20;

DB 설계 패턴
Design Pattern
ERD 설계 프로세스
1
요구사항 분석 — 엔티티 식별 (명사 추출)
2
관계 정의 — 카디널리티 결정 (1:1, 1:N, N:M)
3
속성 정의 — 필수/선택, 도메인 타입 지정
4
정규화 → 필요 시 성능 반정규화
소프트 삭제 (Soft Delete)
  • deleted_at TIMESTAMP NULL 컬럼 추가
  • 조회 시 WHERE deleted_at IS NULL 필수
  • 감사 이력 보존 & 복구 가능, 파티셔닝 고려 필요
이력 관리 패턴
  • Temporal Table — valid_from / valid_to로 시간 관리
  • Audit Log Table — 변경 전후 값을 별도 테이블에 기록
  • Event Sourcing — 상태 대신 이벤트만 저장, 현재 상태는 재생
08
대용량 트래픽 & 확장 전략
Scale
Read Replica (복제)
  • Master 쓰기 전용 / Replica 읽기 전용으로 분리
  • 비동기 복제 → 복제 지연(Replication Lag) 발생 가능
  • 읽기 80~90% 트래픽 분산 효과
  • 재고 차감 등 강한 일관성 필요 시 Master 직접 조회
샤딩 (Sharding)
  • 수평 샤딩 — 동일 스키마, 데이터를 여러 DB에 분산
  • 샤드 키 — user_id % N, 지역, 날짜 등으로 라우팅
  • Cross-shard JOIN 불가, 분산 트랜잭션 복잡
  • 핫스팟 주의 → 균등 분산 설계 필수
캐싱 전략
  • Look-aside (Cache-aside) — 캐시 미스 시 DB 조회 후 캐시 적재 (가장 일반)
  • Write-through — DB와 캐시 동시 쓰기
  • Write-back — 캐시만 쓰고 나중에 DB 반영 (유실 위험)
  • TTL 설정, Cache Stampede 방지 (Lock / Jitter)
Connection Pool
  • DB 커넥션 생성 비용이 크므로 풀로 재사용
  • HikariCP — Spring Boot 기본 (MySQL 기준 권장 10~20)
  • pool_size = (core_count × 2) + effective_spindle_count
  • 커넥션 고갈 → timeout, 쿼리 슬로우 함께 확인
파티셔닝
  • Range — 날짜/ID 범위로 파티션 분리 (로그 테이블)
  • List — 특정 값(지역코드 등)으로 분리
  • Hash — 균등 분산
  • 파티션 프루닝 — WHERE에 파티션 키 포함 시 스캔 최소화
09
SQL vs NoSQL
Comparison
항목 SQL (RDB) NoSQL
스키마 고정 스키마, 엄격한 타입 유연한 스키마 (Schema-less)
쿼리 표준 SQL, 복잡한 JOIN 가능 DB마다 고유 API, JOIN 제한
확장 수직 확장 위주, 샤딩 복잡 수평 확장에 최적화
일관성 ACID 완전 지원 BASE (Basically Available, Eventually Consistent)
적합 케이스 금융, 주문, 복잡한 비즈니스 로직 대규모 비정형 데이터, 고속 읽기/쓰기
Document (MongoDB)
  • JSON/BSON 문서 단위 저장
  • 중첩 구조, 배열 자유롭게 표현
  • 컨텐츠, 카탈로그, CMS에 적합
Key-Value (Redis)
  • 초고속 인메모리, 단순 구조
  • 세션, 캐시, 분산락, 리더보드
  • TTL 기반 자동 만료 지원
Wide-column (Cassandra)
  • 파티션 키 기반 분산, 쓰기 최적화
  • IoT, 시계열, 로그 대량 적재
  • 쿼리 패턴에 맞게 테이블 설계

RDBMS 비교
항목 MySQL PostgreSQL Oracle
라이선스 GPL / 상용(Oracle) PostgreSQL License (자유) 상용 (Enterprise)
기본 격리 수준 REPEATABLE READ READ COMMITTED READ COMMITTED

적합 케이스 웹 서비스, 스타트업, 빠른 읽기 중심 복잡한 쿼리, GIS, 분석, 오픈소스 엔터프라이즈 대형 금융·ERP, 고가용성 필수 환경
MySQL 특징
  • InnoDB — MVCC 기반, 클러스터드 인덱스 구조
  • 읽기 성능 최적화, 웹 스택(LAMP)에서 검증
  • binlog 기반 복제 — 단순하고 안정적
  • 8.0에서 윈도우 함수·CTE 추가, 격차 축소
  • AWS RDS·Aurora에서 가장 많이 사용
PostgreSQL 특징
  • MVCC + 모든 격리 수준 완벽 지원
  • JSONB · 배열 · hstore 등 다양한 타입
  • PostGIS — 공간 데이터(GIS) 사실상 표준
  • Extension 시스템 — pg_vector, TimescaleDB 등
  • SQL 표준 준수율이 가장 높음
Oracle 특징
  • RAC — 다중 인스턴스가 공유 스토리지 접근
  • Undo Tablespace로 읽기 일관성 강력 보장
  • Hint 문법 — 옵티마이저 직접 제어 가능
  • Flashback — 과거 시점 데이터 조회·복구
  • 라이선스 비용 높음, 대기업·금융 중심
선택 기준 요약 — 스타트업·웹 서비스 → MySQL로 빠르게 시작. 복잡한 쿼리·GIS·JSON 분석 → PostgreSQL. 고가용성이 최우선이고 비용 제약이 없는 대형 시스템 → Oracle.

면접 빈출 Q&A
Interview
질문 핵심 답변 포인트
인덱스가 있는데 왜 느린가요? 풀스캔이 유리한 경우(카디널리티 낮음), 함수 감싸기, 복합인덱스 선두 컬럼 미사용, 인덱스 통계 오래됨 → ANALYZE 실행
트랜잭션 격리 수준을 어떻게 선택하나요? READ COMMITTED(일반 서비스) vs REPEATABLE READ(재조회 일관성 필요 시). SERIALIZABLE은 성능 부담 커서 특수 케이스만
N+1 문제란 무엇이고 어떻게 해결하나요? 1번 쿼리로 N개 결과 → N번 추가 쿼리 발생. JOIN 또는 IN 절 Batch, JPA에서는 Fetch Join / @BatchSize / EntityGraph 사용
대용량 테이블 조회 성능을 높이려면? 인덱스 최적화, 커버링 인덱스, 파티셔닝, 캐싱(Redis), Read Replica, 쿼리 재작성, OFFSET→커서 페이징
샤딩과 파티셔닝의 차이는? 파티셔닝 — 단일 DB 내 논리/물리 분할. 샤딩 — 여러 물리 DB에 분산. 샤딩은 scale-out, 파티셔닝은 관리/성능 목적
MVCC란? 쓰기 시 이전 버전을 Undo Log에 유지. 읽기는 스냅샷으로 처리 → Lock 없이 동시성 향상. InnoDB REPEATABLE READ의 기반
CAP 정리를 설명하세요 분산 시스템은 Consistency, Availability, Partition Tolerance 3가지 중 2가지만 동시에 보장 가능. P는 피할 수 없으므로 CP(MySQL, ZooKeeper) vs AP(Cassandra, DynamoDB) 선택
EXPLAIN ANALYZE Slow Query Log MVCC WAL(Write-Ahead Log) Buffer Pool Redo / Undo Log 두 단계 커밋(2PC) Saga Pattern CDC(Change Data Capture)
