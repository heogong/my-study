03 트러블슈팅 사례 DB Connection Pool
🔴 Problem
  • DB Connection Pool 기본값 10개
  • 신규 서비스: DB 읽기/쓰기 + API 검색이 단일 트랜잭션
  • API 검색 지연 → 커넥션 선점 장기화
  • 동시 요청 증가 → Pool 고갈 → DB 연결 오류
  • 서버 다운
→
🟡 Root Cause
  • 하나의 트랜잭션이 DB 작업 + 외부 API 검색을 모두 포함
  • 외부 API 지연 동안 DB 커넥션을 점유
  • Pool 크기 확장은 근본 해결책이 아님
  • DB 작업과 API 검색 로직 분리 필요
→
🟢 Solution
  • DB 읽기/쓰기 ↔ API 검색 로직 분리
  • AWS SQS 도입으로 비동기 처리
  • 클라이언트 요청 → 큐에 적재
  • 큐에서 순차 폴링 → DB 작업 처리
  • 커넥션 과점유 방지 + 안정성 확보

05 마이그레이션 프로세스 Zero Downtime
1
신규 서비스 VPC 구성 & ALB 설정 인프라
신규 서비스를 위한 VPC 새로 구성. 외부 엔드포인트 접근을 위한 ALB 설정으로 신규 서비스 트래픽 라우팅 준비.
2
기존 서비스에 Spring Cloud Gateway 구성 라우팅
기존 서비스 내부에 Spring Cloud Gateway 구성. ALB 라우팅 규칙으로 특정 API 요청을 Gateway로 전달하도록 설정.
3
점진적 마이그레이션 카나리
Gateway에서 특정 헤더·필드값 기반으로 트래픽 구분. 일부 트래픽만 신규 ALB 엔드포인트로 라우팅 → 단계적으로 비율 증가. 기존 시스템 ↔ 신규 시스템 병렬 운영으로 리스크 분산.
4
로그 모니터링 & 롤백 대응 안전망
OpenSearch Service로 로그 실시간 모니터링. 예상치 못한 오류 발생 시 → ALB 라우팅 규칙을 원래 상태로 복구. 트래픽을 기존 서비스로 즉시 롤백 가능.
5
완전한 마이그레이션 완료 완료
점진적 테스트·롤백 과정을 거쳐 모든 트래픽 신규 서비스로 전환. 기존 서비스 종료 또는 필요 부분만 유지. Spring Cloud Gateway를 신규 내부 서비스로 이전 → 내부 서비스 라우팅으로 활용.
마이그레이션 핵심 원칙
  • 점진적 전환 — 한 번에 모두 바꾸지 않음
  • 헤더/필드 기반 트래픽 분기로 세밀한 제어
  • 롤백 플랜 항상 준비 — ALB 규칙 복원
  • 실시간 모니터링으로 이상 신호 즉시 감지
Spring Cloud Gateway 역할 변화
  • 마이그레이션 중: 기존 → 신규 트래픽 분기 포인트
  • 마이그레이션 후: 신규 서비스 내부 라우팅
  • 동일 컴포넌트가 전환 도구 → 운영 도구로 자연스럽게 역할 이동
포인트: Gateway를 마이그레이션 완료 후에도 버리지 않고 내부 라우팅에 재활용
마이그레이션 결과
무중단 배포 + 점진적 전환으로 리스크 최소화. 예상치 못한 이슈 발생 시 빠른 롤백으로 서비스 영향 최소화. 안정적인 신규 서비스 롤아웃 완료.
06 클라우드 전환 전략 (6R) Migration Strategy
Rehost
Lift & Shift
  • 그대로 AWS로 올림
  • 빠른 이전, 코드 수정 불가 시
Replatform
Lift & Reshape
  • 약간 최적화 후 이전
  • DB → RDS, WAS → Beanstalk
  • EC2 → ECS 전환 시 적용
Refactor
Re-architect
  • 클라우드 네이티브로 재설계
  • MSA 전환, 성능/확장성 개선
  • Python → Java + MSA 전환 시 적용
Repurchase
Drop & Shop
  • SaaS로 교체
  • CRM → Salesforce
Retire
폐기
  • 안 쓰는 시스템 정리
  • 불필요 리소스 제거
Retain
Revisit
  • 일단 안 움직임
  • 규제/의존성 문제
  • 나중에 재검토
실무 적용: EC2 → ECS 마이그레이션 시 Replatform + Refactor 병행 — 핵심 서비스는 Python → Java 언어 전환(Refactor), 보조 서비스는 EC2 → ECS(Replatform)
마이그레이션 Assessment
  • 공개 IP 구성 → 보안 취약점 식별
  • 오토스케일링 미구성 → 고정 비용 문제 파악
  • 배포 복잡도 → 빈번한 배포 어려움 확인
  • 인프라팀 인터뷰 + 코드 분석 + 트래픽 패턴 병행
핵심: Assessment = 현황 파악 + 문제점 정의 → 전환 전략 결정의 인풋
무중단 전환 핵심
  • VPC 피어링 불가 (CIDR 충돌) → SCG 우회
  • 기존 환경에 Spring Cloud Gateway 먼저 설치
  • ALB 리스너 규칙으로 경로별 트래픽 제어
  • 검증 완료 후 도메인 전환으로 최종 스위칭
결과: SCG 점진적 전환 → 도메인 스위칭으로 무중단 마이그레이션 완료
07 아키텍처 심화 설계 ZTA · MSA · Gateway
ZTA — Zero Trust Architecture
"Never Trust, Always Verify"
  • 서비스 간 mTLS 인증 적용
  • IAM 최소권한 원칙 — 서비스별 권한 분리
  • AWS WAF 화이트리스트 — 허가된 IP만 허용
  • VPC + 서브넷 + 보안그룹 네트워크 세그멘테이션
핵심: 내부망도 신뢰 불가 — 모든 요청을 검증하는 보안 설계 철학
MSA 서비스 간 통신 설계
  • 동기 → REST API (일반 서비스 통신)
  • 비동기 → AWS SQS (주소검색 등 지연 허용)
  • 요청/검증 분리 → API마다 스레드 과생성 방지
  • SQS 도입 → 데이터 유실 방지 + 안정성 확보
원칙: 동기는 REST, 비동기는 SQS — 처리 특성에 맞게 통신 방식 분리
Spring Cloud Gateway 실무
  • 신세계아이앤씨: 온프레미스 → 클라우드 Canary 전환
  • CODE: EC2 → ECS 무중단 라우팅 제어
  • ALB 뒤에 SCG — 경로 기반 라우팅 설계
  • 가중치 기반 트래픽 분산으로 점진적 전환
포인트: SCG 두 번 실무 적용 — Canary 배포 + 마이그레이션 무중단 트래픽 제어

09 MSA / 서비스 분리 MSA Transition
INTERVIEW Q.
MSA 전환 시 가장 어려운 점이 뭐였고, 어떻게 해결했나요?
전환 배경 & 과정
  • 빗썸시스템즈 — Python FastAPI → Java Spring Boot 전환
  • MSA 구조 확립을 동시에 진행
  • DB 단일 구조 유지 → 데이터 정합성보다 서비스 간 인터페이스 정의가 핵심 난제
  • 기존 Python 서비스와 병행 운영하며 점진적 트래픽 이전
핵심 해결 전략
  • API 스펙 먼저 확정 → OpenAPI 문서화 선행
  • 서비스별 독립 배포 가능한 구조로 순차 전환
  • 언어 전환 리스크 → 병행 운영으로 분산
  • 배포 독립성 확보 후 단계적 트래픽 이전으로 안정성 확보
핵심 한 줄 요약: 언어 전환 + MSA 동시 진행 → API 스펙 선행 + 병행 운영으로 리스크 분산

13 보안 / 인증 mTLS · Travel Rule
INTERVIEW Q.
mTLS 기반 거래소 연동은 어떻게 설계했나요?
mTLS 개요
  • Travel Rule — 국내외 거래소 간 메시지 송수신
  • 양방향 신원 검증 필수 요건
  • 서버 · 클라이언트 양쪽 인증서 교환으로 상호 인증
  • 단방향 TLS 대비 중간자 공격 완전 차단
구현 상세
  • 거래소별 인증서 발급 · 교환
  • Spring Boot SSLContext 커스텀 구성
  • Keystore: 자신의 인증서 · 개인키 보관
  • Truststore: 상대방 인증서 등록 · 검증
결과
  • 신규 거래소 연동 완료 건수 40% 향상
  • 양방향 인증으로 보안 신뢰성 확보
  • 중간자 공격 위험 제거
  • 확장 가능한 거래소 연동 구조 정립
핵심 한 줄 요약: mTLS = 양방향 인증서 검증 → 중간자 공격 차단 → Keystore/Truststore 관리
