APM & 프로파일링
Observability
APM 도구
  • Pinpoint — 네이버 오픈소스, Java Agent 기반 분산 추적
  • Datadog APM — SaaS, 인프라·로그·트레이스 통합

  • BLOCKED 상태 스레드 → 데드락 또는 락 경합
  • WAITING 상태 과다 → 스레드풀 고갈 의심
  • FastThread, IBM Thread Analyzer로 시각화 분석
힙 덤프 분석
  • jmap -dump:format=b,file=heap.hprof <PID>
  • -XX:+HeapDumpOnOutOfMemoryError 옵션으로 자동 생성
  • Eclipse MAT으로 Dominator Tree, Leak Suspects 분석
  • Retained Heap이 큰 객체 → 누수 원인 후보
상황 진단 도구 확인 포인트

메모리 증가 / OOM 힙 덤프, GC 로그 Retained Heap 큰 객체, ThreadLocal 누수

커넥션 고갈 커넥션 풀 모니터링, 스레드 덤프 미반환 커넥션, 트랜잭션 장시간 유지

트래픽 급증 대응
Resilience
Rate Limiting / 스로틀링
  • 클라이언트별 초당 요청 수 제한 (Token Bucket 알고리즘)
  • Spring Cloud Gateway, Bucket4j, Redis 기반 구현
  • 429 Too Many Requests + Retry-After 헤더 응답
  • IP·사용자·API Key 단위로 정책 분리
서킷 브레이커 (Resilience4j)
  • CLOSED → 정상 호출 통과
  • OPEN → 실패율 임계치 초과 시, 즉시 fallback 반환
  • HALF_OPEN → 일부 호출 허용, 복구 여부 판단
  • Bulkhead로 스레드/동시성 제한, TimeLimiter로 타임아웃 제어
트래픽 급증 체크리스트
  • 수평 확장(Scale Out) — 오토스케일링 정책 사전 설정
  • DB 커넥션 풀 크기 검토 (HikariCP maxPoolSize)
  • 비동기 처리 전환 — 큐(Kafka, SQS)로 부하 분산
  • CDN으로 정적 자원 오프로드
  • 읽기 집중 API에 캐시 레이어 추가
  • 불필요한 동기 외부 호출 타임아웃 단축
03
캐시 전략
Cache
Cache Aside (Lazy)
  • 읽기 시 캐시 미스 → DB 조회 → 캐시 저장
  • 가장 일반적인 패턴, 구현 간단
  • 캐시 미스 직후 Cache Stampede 주의
  • 해결: 뮤텍스 락 또는 확률적 조기 갱신(PER)
Write Through
  • 쓰기 시 DB와 캐시 동시에 업데이트
  • 캐시와 DB 항상 동기화 → 읽기 일관성 높음
  • 쓰기 지연 증가, 사용되지 않는 데이터도 캐시됨
Write Behind (Async)
  • 쓰기 시 캐시만 업데이트 후 비동기로 DB 반영
  • 쓰기 성능 최고, 배치 병합으로 DB 부하 감소
  • 캐시 장애 시 데이터 유실 위험 (내구성 낮음)
이슈 원인 해결책
Cache Stampede 동시에 캐시 만료 → 다수 요청이 DB 동시 접근 뮤텍스 락, PER(Probabilistic Early Recomputation), 캐시 TTL 지터
Cache Penetration 존재하지 않는 키 반복 요청 → DB 직접 타격 Null 값도 캐시 저장, Bloom Filter로 사전 필터링
Cache Avalanche 대량 캐시 동시 만료 → DB 폭주 TTL에 랜덤 지터(jitter) 추가, 핫 데이터는 무기한 캐시
04
장애 포스트모텀
Incident
포스트모텀 원칙

포스트모텀은 비난 없는(Blameless) 문화가 핵심입니다. 개인이 아닌 시스템과 프로세스의 취약점을 찾아 재발을 방지합니다.

1
타임라인 작성 — 장애 감지 시각, 각 대응 행동, 복구 완료까지 시간 순으로 정확히 기록. 로그·알람·Slack 메시지를 증거로 활용
2
근본 원인 분석 (5 Whys) — "왜?"를 반복해 표면적 증상이 아닌 근본 원인 도달. 단일 원인보다 복합 요인인 경우가 많음
3
영향도 정리 — 영향받은 사용자 수, 서비스 다운 시간(MTTR), 비즈니스 손실 추정
4
재발 방지 액션 — 단기(즉각 핫픽스)와 장기(아키텍처 개선) 액션을 구분, 담당자와 기한 명시
5
공유 및 학습 — 팀 전체에 공개, 런북(Runbook) 업데이트, 알람 임계치 재설정
MTTR vs MTTD — MTTD(Mean Time To Detect)는 장애 감지까지 걸린 시간, MTTR(Mean Time To Recover)은 복구까지 걸린 시간. 알람 민감도를 높여 MTTD를 줄이는 것이 선행 과제입니다.

JVM 성능 튜닝
JVM
힙 크기 설정
  • -Xms(초기) = -Xmx(최대) 로 설정 → GC 재조정 오버헤드 제거
  • 일반적으로 사용 가능 물리 메모리의 50~75% 할당
  • 컨테이너 환경: -XX:MaxRAMPercentage=75 로 동적 설정
  • Old Gen 크기: Full GC 후 Live Set의 2~3배 확보
GC 알고리즘 선택
  • G1GC (Java 9+ 기본) — 힙 4GB 이상, 응답시간 중시
  • ZGC — STW 1ms 이하, 대용량 힙(수백GB) 특화
  • Shenandoah — ZGC와 유사, Red Hat 주도
  • ParallelGC — 배치 처리, 처리량(Throughput) 중시
# GC 로그 활성화 (Java 9+) -Xlog:gc*:file=/logs/gc.log:time,uptime,level,tags:filecount=5,filesize=20m # G1GC 주요 옵션 -XX:+UseG1GC -XX:MaxGCPauseMillis=200 # 목표 STW 시간 (기본 200ms) -XX:G1HeapRegionSize=16m # 힙이 클수록 큰 리전 크기 권장 -XX:InitiatingHeapOccupancyPercent=45 # OOM 발생 시 자동 힙 덤프 -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/logs/heap.hprof
GC 튜닝의 첫 번째 원칙: GC 로그를 먼저 분석하고, Full GC 빈도·STW 시간·Promotion Failure 여부를 확인한 후 파라미터를 조정합니다. 추측으로 튜닝하지 않습니다.

DB 쿼리 최적화
Database
EXPLAIN 분석 포인트
  • type: ALL → Full Table Scan, 인덱스 추가 필요
  • type: ref / range → 인덱스 사용 (양호)
  • Extra: Using filesort → ORDER BY 인덱스 최적화 필요
  • Extra: Using temporary → GROUP BY 임시 테이블 생성 (주의)
  • rows 값 큰 경우 → 인덱스 선택성 검토
커넥션 풀 튜닝 (HikariCP)
  • 공식 권장: maxPoolSize = (코어수 × 2) + 유효 스핀들 수
  • CPU 4코어 서버: maxPoolSize ≈ 10 (과도한 설정 역효과)
  • connectionTimeout: 30초 (기본), 급증 대응 시 단축 검토
  • leakDetectionThreshold: 미반환 커넥션 감지 (개발 시 2000ms)
-- Slow Query Log 활성화 (MySQL) SET GLOBAL slow_query_log = ON; SET GLOBAL long_query_time = 1; -- 1초 이상 쿼리 기록 SET GLOBAL log_queries_not_using_indexes = ON; -- EXPLAIN ANALYZE (MySQL 8.0+ / PostgreSQL) EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 123 AND status = 'PAID' ORDER BY created_at DESC LIMIT 20;
EXPLAIN ANALYZE Slow Query Log 커버링 인덱스 HikariCP 서킷 브레이커 Cache Stampede 5 Whys Blameless Postmortem G1GC / ZGC async-profiler

실패 경험 — SCG 마이그레이션
Shinsegae I&C
프로젝트 배경

스타벅스 On-Premise → 클라우드 점진적 마이그레이션 (2022.01 – 2022.05)
Spring Cloud Gateway 담당 백엔드 개발자로, Canary 배포를 통한 트래픽 점진 전환을 설계·실행.
기술 스택: Spring Boot · Spring Cloud Gateway · Spring WebFlux · JPA · Elastic Beanstalk · Redis

실패 1 — 성능 테스트 저하
  • 증상: nGrinder 부하 테스트 결과 기존 대비 처리량 현저히 저하
  • 원인: 기존 서비스가 동기 블로킹 방식 → SCG의 Netty 리액티브 런타임과 충돌, 소수 스레드가 점유·대기하며 처리량 급락
  • 해결: Mono / Flux 리액티브 체인으로 전면 전환, 논블로킹 I/O 구조 재개발 → 성능 목표치 달성
  • 교훈: 기술 선택 시 해당 기술의 동작 모델(스레드 모델)을 반드시 함께 이해해야 함
실패 2 — 운영 라우팅 미동작
  • 증상: 테스트 검증 완료 후 운영 배포 → 트래픽이 신규 Cloud가 아닌 On-Premise로만 라우팅
  • 원인 ①: 환경별 yml 분리 미흡 → 테스트용 Canary weight 값이 운영에 그대로 적용, Cloud weight 미설정
  • 원인 ②: Gateway Predicate 조건이 운영 도메인으로 재설정되지 않아 라우팅 매칭 실패 복합 발생
  • 해결: application-prod.yml 환경별 분리, Canary weight 환경변수 외부화, 배포 체크리스트에 라우팅 검증 항목 추가
면접 답변 흐름 — "실패 경험과 극복 방법을 말씀해주세요"
1
상황 설정 — 신세계아이앤씨에서 스타벅스 On-Premise → 클라우드 마이그레이션 프로젝트의 SCG 담당으로 투입, 크게 두 번의 실패를 겪었습니다.
2
실패 1 (성능) — nGrinder 부하 테스트에서 처리량 급락. 코드 리뷰로 동기 블로킹 코드와 Netty 리액티브 런타임의 충돌 확인 → Mono/Flux 리액티브 체인으로 전환해 목표치 달성.
3
실패 2 (라우팅) — 운영 배포 직후 트래픽이 On-Premise로만 흐름. 환경별 yml 미분리 + Predicate 도메인 미설정의 복합 원인 → 환경별 설정 외부화 + 배포 체크리스트 수립.
4
마무리 (전이) — 이 경험을 CODE ECS Fargate 마이그레이션에 적용, 환경별 설정 분리와 점진적 전환 모니터링 프로세스를 처음부터 갖추어 다운타임 0 달성.
꼬리 질문 핵심 답변
리액티브 전환 시 어려웠던 점은? 팀원들의 WebFlux 경험 부재 → Mono/Flux 에러 핸들링 패턴 정리·공유로 러닝커브 단축
Canary 배포 비율은 어떻게 결정했나? 초기 5~10% 시작 → 에러율·응답시간 지표 확인 후 단계적 확대. 롤백 기준(에러율 임계치) 사전 정의
같은 실수를 다시 하지 않은 방법은? 환경별 설정 외부화 + 배포 체크리스트 프로세스화 → CODE에서 동일 방식 적용, 재발 없었음
Spring WebFlux vs MVC 차이는? MVC: 스레드 per request 블로킹 모델 / WebFlux: 이벤트 루프 논블로킹. I/O 대기 구간에서 WebFlux 효율적, CPU 바운드에서는 이점 없음
