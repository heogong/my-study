Q.01 레이어드 아키텍처에서 각 계층의 역할은? ▼
"Presentation → Business → Persistence → DB 순으로 단방향 의존합니다.
Presentation은 요청/응답 처리, Business는 핵심 로직과 트랜잭션, Persistence는 DB 추상화, DB는 실제 데이터 저장을 담당합니다.
각 계층은 바로 아래 계층만 참조해 결합도를 낮춥니다."
Q.02 @Transactional 동작 원리는? ▼
"Spring AOP의 Proxy 패턴으로 동작합니다."
@Transactional이 붙은 메서드 호출 시, Spring이 만든 프록시 객체가 트랜잭션 시작 → 실제 메서드 실행 → 커밋/롤백 순으로 처리합니다.
// 같은 클래스 내 self-call은 프록시를 거치지 않아 트랜잭션 적용 X public class OrderService { @Transactional public void createOrder() { // ✅ 프록시 통과 saveOrder(); // ❌ self-call — 트랜잭션 미적용 } }
Q.03 DI(의존성 주입) 방식 3가지와 권장 방식은? ▼
"생성자 주입, 필드 주입, Setter 주입이 있습니다. 생성자 주입이 권장됩니다."
// ✅ 권장: 생성자 주입 — 불변, 테스트 용이, 순환 의존 감지 private final UserRepository repo; public UserService(UserRepository repo) { this.repo = repo; } // ❌ 비권장: 필드 주입 — 테스트 어렵고 순환 참조 런타임 오류 @Autowired private UserRepository repo;
Q.04 Entity와 DTO를 왜 분리하나요? ▼
"Entity는 DB 테이블과 직접 매핑되어 변경이 민감하고, DTO는 계층 간 데이터 전송 전용입니다."
Entity를 직접 반환하면 불필요한 컬럼 노출, 양방향 순환 참조, API 스펙과 DB 스키마 강결합 문제가 발생합니다.
분리하면 API 변경 시 Entity를 건드리지 않아도 되며, 보안과 유지보수가 향상됩니다.
Q.05 Strategy 패턴을 실무에서 어떻게 적용했나요? ▼
"결제 방식 처리에 Strategy 패턴을 적용했습니다."
interface PaymentStrategy { void pay(int amount); } @Component class CardPayment implements PaymentStrategy { ... } @Component class KakaoPayment implements PaymentStrategy { ... } // PaymentType Enum으로 전략 선택 → if/else 제거, OCP 준수
새로운 결제 수단 추가 시 기존 코드를 수정하지 않고 클래스만 추가하면 되어 확장에 유리합니다.
Q.06 MSA vs 모놀리식 아키텍처 차이는? ▼
"모놀리식은 단일 배포 단위로 개발이 단순하지만, 규모가 커지면 배포·확장이 어렵습니다."
MSA는 서비스별 독립 배포·확장·기술 스택 선택이 가능하지만 네트워크 지연, 분산 트랜잭션, 운영 복잡도가 증가합니다.
초기에는 모놀리식으로 시작해 경계가 명확해지면 MSA로 분리하는 것을 권장합니다.
