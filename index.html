Q.01 JVM 구조를 설명하세요 ▼
"JVM은 ClassLoader, Runtime Data Area, Execution Engine, Native Interface로 구성됩니다."
ClassLoader가 .class 파일을 로드해 메모리에 적재하고, Execution Engine(Interpreter + JIT Compiler)이 바이트코드를 실행합니다.
Runtime Data Area는 Heap(객체), Stack(메서드 프레임), Method Area(클래스 메타·static), PC Register, Native Method Stack으로 나뉩니다.
Q.02 인터페이스와 추상 클래스의 차이는? ▼
"인터페이스는 '무엇을 할 수 있는가(능력)', 추상 클래스는 '무엇인가(본질)'를 표현합니다."
// 인터페이스: 다중 구현 가능, 상태(필드) 없음, IS-ABLE-TO 관계 interface Flyable { void fly(); } interface Swimmable { void swim(); } class Duck implements Flyable, Swimmable { ... } // 추상 클래스: 단일 상속, 상태·공통 구현 공유, IS-A 관계 abstract class Animal { protected String name; // 공통 상태 abstract void sound(); // 서브클래스 구현 강제 }
Java 8+ 이후 인터페이스도 default/static 메서드를 가질 수 있어 차이가 줄었지만, 상태(인스턴스 필드) 보유 여부와 IS-A vs IS-ABLE-TO 의미가 핵심 구분점입니다.
Q.03 equals()와 ==의 차이, hashCode() 계약은? ▼
"==은 참조(주소) 비교, equals()는 값(내용) 비교입니다."
String a = new String("hello"); String b = new String("hello"); a == b // false — 다른 객체 a.equals(b) // true — 값 동일 // hashCode 계약: equals()가 true면 hashCode()도 반드시 같아야 함 // HashMap/HashSet은 hashCode로 버킷 찾고, equals로 동등성 확인
equals를 재정의할 때 hashCode도 반드시 함께 재정의해야 합니다. 그렇지 않으면 HashMap에서 동등한 키가 다른 버킷에 저장되어 조회 실패가 발생합니다.
Q.04 불변 객체(Immutable Object)를 만드는 방법은? ▼
"클래스를 final 선언, 모든 필드를 private final, 생성자에서만 초기화, setter 제거, 방어적 복사를 적용합니다."
public final class Money { private final int amount; private final Currency currency; public Money(int amount, Currency currency) { this.amount = amount; this.currency = currency; } // getter만 제공, setter 없음 // 변경이 필요하면 새 Money 객체 반환 public Money add(int val) { return new Money(amount + val, currency); } }
불변 객체는 Thread-safe, 캐싱 가능, 부작용 없음의 장점이 있어 Value Object와 함수형 스타일 설계에 적합합니다.
Q.05 제네릭(Generics)의 목적과 타입 소거(Type Erasure)란? ▼
"제네릭은 컴파일 타임 타입 안전성을 보장하고 형변환(casting) 코드를 제거합니다."
// 제네릭 없이 — 런타임 ClassCastException 위험 List list = new ArrayList(); list.add("hello"); Integer i = (Integer) list.get(0); // 런타임 오류! // 제네릭 사용 — 컴파일 타임에 타입 오류 감지 List<String> list = new ArrayList<>(); list.add("hello"); String s = list.get(0); // 형변환 불필요
타입 소거(Type Erasure): 컴파일 후 제네릭 타입 정보는 제거되어 런타임에는 Raw Type으로 동작합니다. 따라서 instanceof List<String> 같은 런타임 타입 검사는 불가능합니다.
Q.06 Java에서 동기화(Synchronization) 처리 방법은? ▼
"synchronized 키워드, volatile, java.util.concurrent 패키지 등을 상황에 맞게 사용합니다."
// synchronized — 메서드/블록 단위 상호 배제(mutual exclusion) public synchronized void increment() { count++; } // volatile — 가시성 보장(캐시 미사용, 메인 메모리 직접 읽기) private volatile boolean running = true; // AtomicInteger — CAS(Compare-And-Swap) 기반 락 없는 원자 연산 AtomicInteger count = new AtomicInteger(0); count.incrementAndGet(); // Thread-safe // ReentrantLock — synchronized보다 세밀한 락 제어
volatile은 가시성만 보장(원자성 X), synchronized는 원자성+가시성 모두 보장하지만 성능 비용이 있습니다. 고성능이 필요하면 java.util.concurrent 패키지를 우선 검토합니다.

Q.07 volatile, synchronized, ReentrantLock 의 차이는? ▼
"volatile은 가시성, synchronized는 원자성+가시성, ReentrantLock은 명시적 락 제어입니다."
// volatile — 가시성 보장, 원자성 보장 X private volatile boolean running = true; // 다른 스레드에 즉시 반영 // synchronized — 원자성 + 가시성, 블록 단위 상호배제 public synchronized void increment() { count++; } // ReentrantLock — tryLock, 타임아웃, Condition 지원 ReentrantLock lock = new ReentrantLock(); if (lock.tryLock(1, TimeUnit.SECONDS)) { try { /* 임계구역 */ } finally { lock.unlock(); } }
synchronized는 JVM이 자동으로 락을 해제하지만 ReentrantLock은 반드시 finally에서 unlock() 호출이 필요합니다. 고성능 동시성이 필요하면 AtomicInteger(CAS 기반)를 우선 검토합니다.
Q.08 ThreadLocal 사용 케이스와 메모리 누수 위험은? ▼
"스레드별 독립 저장소로 MDC 로깅, 인증 컨텍스트, 트랜잭션 연결에 활용합니다."
private static final ThreadLocal<User> currentUser = new ThreadLocal<>(); currentUser.set(user); // 저장 User u = currentUser.get(); // 조회 currentUser.remove(); // 반드시 제거! 스레드풀 재사용 시 누수 방지
스레드풀 환경에서 remove() 누락 시 이전 요청 데이터가 다음 요청에 노출됩니다. Spring의 SecurityContextHolder, RequestContextHolder 모두 ThreadLocal 기반입니다.
Q.09 CompletableFuture와 Future의 차이는? ▼
"Future는 블로킹 get()만 지원하지만 CompletableFuture는 논블로킹 콜백 체이닝을 지원합니다."
// Future — get()에서 블로킹 발생 Future<String> f = executor.submit(() -> fetch()); String result = f.get(); // 스레드 블로킹! // CompletableFuture — 논블로킹 체이닝 CompletableFuture.supplyAsync(() -> fetchUser(id)) .thenApply(user -> enrich(user)) .thenAccept(result -> save(result)) .exceptionally(ex -> { log(ex); return null; });
thenCompose()로 비동기 작업 순차 연결, thenCombine()으로 두 결과를 병합합니다. Java 21 Virtual Thread와 함께 사용하면 I/O 집약 서비스에서 특히 효과적입니다.
Q.10 Java 8 ~ 21 주요 변경사항을 설명하세요 ▼
"람다/스트림(8) → 모듈(9) → 텍스트블록(13) → 레코드/봉인클래스(16/17) → 가상스레드(21)"
// Java 8 — 람다, Stream API, Optional, Default Method list.stream().filter(x -> x > 0).map(String::valueOf).collect(Collectors.toList()); // Java 16 — Record (불변 데이터 클래스 간소화) record Point(int x, int y) {} // getter, equals, hashCode, toString 자동 생성 // Java 17 — Sealed Class (상속 허용 클래스 제한) sealed interface Shape permits Circle, Rectangle {} // Java 21 — Virtual Thread (경량 스레드, 블로킹 I/O 혁신) Thread.ofVirtual().start(() -> handleRequest());
LTS 버전은 8, 11, 17, 21입니다. Virtual Thread는 수백만 개의 경량 스레드를 생성할 수 있어 I/O 집약 서비스에서 스레드풀 튜닝 부담을 크게 줄여줍니다.
