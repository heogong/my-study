Full GC (Fallback) STW (최장) Evacuation 실패 시 Serial Old GC로 폴백 → 반드시 피해야 함

단골 질문 & 모범 답변
Q.01 GC가 무엇이고 왜 필요한가요? ▼
"GC(Garbage Collector)는 JVM이 더 이상 참조되지 않는 객체의 메모리를 자동으로 회수하는 메커니즘입니다.
Java는 개발자가 직접 메모리를 해제하지 않고 GC에 위임함으로써 메모리 누수와 dangling pointer 문제를 줄입니다.
단점은 GC가 동작하는 동안 STW(Stop-The-World)가 발생해 응답 지연이 생길 수 있다는 점입니다."
Q.02 Minor GC vs Major GC vs Full GC 차이는? ▼
"Minor GC는 Young Generation(Eden+Survivor)을 대상으로 하며 빠릅니다.
Major GC는 Old Generation을 청소하며 STW가 깁니다.
Full GC는 힙 전체(Young+Old+Metaspace)를 정리하며 가장 오래 걸립니다.
실서비스에서 Full GC가 자주 발생하면 힙 크기 조정이나 GC 알고리즘 변경이 필요합니다."
Q.03 G1 GC를 쓰는 이유는? ▼
"기존 CMS는 Compaction이 없어 메모리 단편화가 발생하고, Parallel GC는 STW가 깁니다.
G1은 힙을 Region 단위로 나눠서 가비지가 많은 Region을 우선 수거(Garbage-First)하므로 STW 시간이 예측 가능합니다.
Java 9부터 기본 GC이며, 대규모 힙(4GB 이상)에서 응답성과 처리량의 균형이 좋습니다."
Q.04 STW(Stop-The-World)란? ▼
"GC가 실행되는 동안 JVM이 모든 애플리케이션 스레드를 일시 정지시키는 현상입니다.
이 시간이 길수록 응답 지연(latency)이 커집니다.
ZGC, Shenandoah는 STW를 1ms 미만으로 줄이기 위해 대부분의 작업을 애플리케이션 스레드와 동시에 처리합니다."
Q.05 메모리 누수(Memory Leak)가 Java에서도 발생하나요? ▼
"GC가 있어도 참조가 남아있으면 메모리가 회수되지 않아 누수가 발생합니다.
대표적인 원인으로는 static 컬렉션에 객체를 계속 추가, 이벤트 리스너 미해제, ThreadLocal 미제거, 캐시 무한 성장 등이 있습니다.
VisualVM, JProfiler, Eclipse MAT 같은 도구로 Heap Dump를 분석해 진단합니다."
Q.06 GC 튜닝 시 가장 먼저 확인할 것은? ▼
"먼저 GC 로그를 수집해 STW 빈도와 시간을 확인합니다.
Full GC가 잦다면 힙 크기를 늘리거나 Old Gen 비율을 조정합니다.
Minor GC가 너무 잦다면 Young Gen 크기를 늘립니다.
응답 지연이 목표라면 G1 → ZGC로 전환을 검토하고, 처리량이 중요하면 Parallel GC가 적합할 수 있습니다.
기본 원칙은 '측정 후 튜닝'입니다."
