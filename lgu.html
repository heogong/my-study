1. 예상 질문 & 답변 (핵심만)

[마이그레이션 전략]

Q. 온프레미스 → AWS 전환 전략(6R)을 어떻게 적용했나요?

CODE에서 EC2 기반 서비스를 ECS Fargate로 전환할 때, 서비스별로 전략을 달리했습니다.
핵심 서비스(트래블룰)는 Python→Java 언어 전환과 인프라 전환을 동시에 진행했기 때문에 Refactor에 가까웠고, 보조 서비스는 구조 변경 없이 컨테이너화하는 Replatform 전략이었습니다.
한 번에 전부 바꾸지 않고, Spring Cloud Gateway로 트래픽을 점진적으로 전환하면서 리스크를 분산했습니다.

한 줄 요약: Refactor + Replatform 병행, SCG 기반 점진적 전환으로 무중단 구현

Q. 전환 전 Assessment는 어떻게 했나요?

자동화 툴보다는 직접 인프라 현황과 코드 레벨을 분석했습니다.
당시 발견한 문제는 세 가지였습니다: 공개 IP 구성으로 인한 보안 취약점, 오토스케일링 미구성으로 인한 고정 비용 낭비, 배포 프로세스 복잡도로 인한 배포 주기 문제.
이 문제들을 인프라팀 인터뷰 + 코드 분석 + 트래픽 패턴 확인으로 도출하고, 마이그레이션 전략의 근거로 삼았습니다.

한 줄 요약: 인프라 현황 파악 + 문제 정의 → 전략 선택의 인풋

Q. 다운타임 없이 어떻게 전환했나요?

VPC 피어링이 CIDR 충돌로 불가능한 상황이었습니다.
그래서 기존 환경에 SCG를 먼저 설치하고, ALB 리스너 규칙으로 특정 경로의 트래픽만 신규 환경으로 라우팅하는 1단계를 먼저 진행했습니다.
검증 완료 후 도메인 전환으로 전체 트래픽을 신규 VPC로 이전하는 2단계 구조로, 결과적으로 다운타임 0으로 완료했습니다.

[아키텍처 설계]

Q. ZTA를 실무에서 어떻게 적용했나요?

ZTA 전용 프로젝트는 없었지만, CODE에서 네 가지 방식으로 원칙을 실무 적용했습니다.
서비스 간 mTLS 인증으로 내부망도 검증하는 구조, IAM 최소권한 + ECS Task Role 단위 권한 분리, AWS WAF 화이트리스트로 허가된 IP만 접근 허용, VPC + 서브넷 + 보안그룹으로 네트워크 세그멘테이션을 구성했습니다.

한 줄 요약: mTLS + IAM 최소권한 + WAF 화이트리스트 + 네트워크 세그멘테이션 = 실질적 ZTA 적용

Q. 동기/비동기 통신을 어떻게 나눠서 설계했나요?

Python FastAPI 모놀리스를 Spring Boot MSA로 전환하면서, 동기는 REST, 비동기는 SQS로 분리했습니다.
주소검색처럼 요청마다 외부 API를 호출하는 구간은 API 요청마다 스레드가 생성되는 문제가 있었는데, SQS로 분리해서 요청과 검증을 비동기로 처리하고 데이터 유실도 방지했습니다.

Q. API Gateway 설계 경험이 있나요?

Spring Cloud Gateway를 두 번 실무에서 사용했습니다.
신세계아이앤씨에서는 스타벅스 Canary 트래픽 전환용, CODE에서는 마이그레이션 중 무중단 라우팅 제어용이었습니다.
두 경우 모두 ALB 뒤에 SCG를 두고, 경로 기반 라우팅 + 가중치 기반 트래픽 분산을 구성했습니다.

Q. ECS와 EKS 트레이드오프를 설명해보세요.

ECS는 진입장벽이 낮고 AWS 친화성이 높지만 커스터마이징이 제한적입니다.
EKS는 확장성과 멀티클라우드 유연성이 높지만 운영 복잡도가 높습니다.
CODE에서는 팀 규모와 운영 부담을 고려해 ECS Fargate를 선택했고, 이수시스템에서는 멀티 서비스 확장성과 NCP 환경 특성으로 Kubernetes를 선택했습니다.
선택 기준은 팀 역량, 운영 부담, 멀티클라우드 필요성입니다.

[실패 경험]

Q. 실패 경험을 말해보세요.

신세계아이앤씨 스타벅스 마이그레이션에서 두 번의 실패:

실패 1 — SCG 도입 후 성능 저하

nGrinder 부하 테스트에서 처리량이 현저히 낮게 나왔습니다.
원인은 기존 서비스가 동기 블로킹 방식인데, SCG의 Netty 기반 리액티브 런타임과 충돌해 스레드가 점유되는 문제였습니다.
Mono/Flux 리액티브 체인으로 전면 전환해서 성능 목표를 달성했습니다.

실패 2 — 운영 배포 후 라우팅 미동작

테스트 서버 검증 완료 후 운영 배포했는데, 트래픽이 클라우드가 아닌 온프레미스로만 흘렀습니다.
원인은 두 가지 복합: 환경별 yml 미분리로 Canary weight 미설정, Gateway Predicate가 운영 도메인 기준으로 재설정 안 됨.
application-prod.yml 분리 + Canary weight 환경변수 외부화 + 배포 체크리스트 추가로 해결했습니다.

배운 것: 기술 선택 시 동작 모델 이해 필수 / 환경별 설정 명시적 관리 없으면 검증된 내용도 운영에서 깨진다.

[리딩/협업 경험]

Q. 리딩 또는 PM 경험이 있나요?

이수시스템에서 Claude Enterprise 도입 파일럿의 전사 PM을 맡았습니다.
전사 아이디어 공모 → 수십 개 수집 → 기준(측정 가능성/구현 가능성/전략 정합성) 직접 설계 → 4개 과제 선정 → 다부서 TF 운영 → CEO 직보 구조였습니다.
공식 권한이 없는 구조였기 때문에, CEO 보고 가시성과 인센티브 구조를 설계해 자발적 참여를 이끌었습니다.
서버 보안 점검 자동화 과제는 수동 20대에서 250대 전수 자동화로 전환, 외부 도구 구독 비용도 함께 절감했습니다.

[문서화/커뮤니케이션]

Q. 비개발 이해관계자와 협업한 경험이 있나요?

AX 파일럿 PM으로 CEO 보고 자료를 직접 작성하고 보고했습니다.
ROI 구조를 명확히 하기 위해 Before 수치(현재 공수)를 먼저 확보하고, 도입 후 절감 목표와 대비해서 숫자로 설명하는 구조를 만들었습니다.
기술 맥락이 없는 경영진에게 효과를 설명할 때는, 기술 용어 대신 비용·시간·리스크 언어로 번역하는 것이 핵심이었습니다.
