Rate Limiting 전략
  • Token Bucket — 토큰 충전 후 소비. 버스트 허용
  • Leaky Bucket — 일정 속도로 처리. 버스트 차단
  • Fixed Window — 시간 창 단위 카운팅. 경계 문제 발생
  • Sliding Window — 가장 정확. 구현 복잡도 높음

  • Graceful Shutdown — 처리 중 요청 완료 후 종료
  • Connection Draining — 기존 연결 유지, 신규 차단
Sticky Session 문제

  • JWT로 무상태(Stateless) 설계 선호
03
다단계 캐싱 전략
Caching
캐시 계층 구조 (위로 갈수록 빠름, 용량 작음)
① 브라우저 캐시 Cache-Control, ETag — ~ns 접근
② CDN 캐시 Cloudflare, CloudFront — 엣지 서버 캐시
③ 애플리케이션 로컬 캐시 Caffeine, Guava Cache — JVM Heap, μs
④ 분산 캐시 (Redis/Memcached) 서버 간 공유, ms 이하 — 가장 중요
⑤ DB 캐시 (Query Cache / Buffer Pool) InnoDB Buffer Pool, PostgreSQL Shared Buffers
캐시 갱신 전략
전략동작특징
Cache-Aside Miss 시 DB 조회 후 캐시 저장 가장 일반적 권장
Write-Through 쓰기 시 캐시+DB 동시 갱신 일관성↑, 쓰기 지연↑
Write-Behind 캐시 먼저 쓰고 DB는 비동기 쓰기 성능↑, 유실 위험
Read-Through 캐시가 자동으로 DB 조회 캐시가 데이터소스 역할
Refresh-Ahead 만료 전 자동 갱신 응답 지연 없음, 복잡도↑
캐시 장애 패턴
  • Cache Stampede (Thundering Herd)
    캐시 만료 순간 다수 요청이 DB에 동시 접근
    → Mutex Lock, PER(Probabilistic Early Recomputation)
  • Cache Penetration (캐시 관통)
    존재하지 않는 키 반복 요청 → DB 직접 접근
    → Bloom Filter로 사전 차단, Null 값 캐싱
  • Cache Avalanche (캐시 붕괴)
    대량 키가 동시 만료 → DB 과부하
    → TTL에 랜덤 지터(Jitter) 추가
Redis 핵심 자료구조
  • String — 카운터, 세션, 단순 캐시
  • Hash — 사용자 프로필, 객체 저장
  • List — 메시지 큐, 피드 타임라인
  • Set — 유니크 방문자, 태그
  • ZSet — 랭킹, 우선순위 큐
  • HLL — 근사 카운팅 (UV 집계)
Redis 운영 패턴
  • Cluster Mode — 16384 슬롯 샤딩, 수평 확장
  • Sentinel — 마스터 감시 + 자동 페일오버
  • Pipeline — 다중 명령 배치 전송, RTT 절감
  • Lua Script — 원자적 복합 연산
  • Pub/Sub — 이벤트 브로드캐스트
캐시 키 설계 원칙
  • 네임스페이스 구분: user:123:profile
  • 버전 포함: v2:product:456
  • TTL 반드시 설정 (메모리 누수 방지)
  • 너무 큰 Value는 분할 저장
  • Hot Key 분산: 키에 suffix 붙여 샤드 분리

Kafka 핵심 개념
  • Topic — 메시지 카테고리. 파티션으로 분할
  • Partition — 병렬 처리 단위. Consumer 수와 매핑
  • Consumer Group — 같은 그룹 내 파티션 분산 소비
  • Offset — 소비 위치 커밋. 재처리/재시작 가능
  • Replication Factor — 파티션 복제 수 (장애 복구)
  • ISR — 동기화된 복제본 집합
At-least-once vs Exactly-once: 기본은 At-least-once. Exactly-once는 Idempotent Producer + Transactional API 조합.
Dead Letter Queue (DLQ)
  • 처리 실패 메시지를 별도 큐로 격리
  • 재시도 횟수 초과 후 DLQ로 이동
  • 장애 원인 분석 후 재처리
  • 알람 연동으로 모니터링
Outbox Pattern
  • DB 트랜잭션과 메시지 발행을 원자적으로 처리
  • outbox 테이블에 이벤트 저장 → 별도 프로세스가 MQ 발행
  • 서비스 장애 시 메시지 유실 방지
  • CDC (Debezium) 연동 가능
SAGA 패턴
  • MSA 분산 트랜잭션 처리
  • Choreography — 이벤트로 서비스 간 연쇄 처리
  • Orchestration — 중앙 오케스트레이터가 제어
  • 실패 시 보상 트랜잭션(Compensating) 실행

  • 수평 샤딩 (Horizontal) — 행 단위 분산
    예: user_id % 4 → shard 0~3
  • 범위 기반 — user_id 1~1M → shard1
    핫스팟(Hot Spot) 위험
  • 해시 기반 — 균등 분산. 범위 쿼리 어려움
  • 디렉토리 기반 — 매핑 테이블로 유연 관리. 조회 오버헤드
  • 크로스 샤드 쿼리 문제: 집계, JOIN이 어려움
    → CQRS + 별도 읽기 모델(ES/검색엔진)

Interview Q&A
1초에 100만 요청을 처리하는 시스템을 설계해 주세요.
계층별 접근:

①네트워크 레이어 — DNS 기반 글로벌 로드 밸런싱(Anycast), CDN으로 정적 자산 엣지 캐싱. 실질적으로 오리진 도달 트래픽을 대폭 감소.

②애플리케이션 레이어 — API Gateway에서 Rate Limiting + 인증 처리. 비즈니스 서비스는 MSA로 분리해 독립 확장. Kubernetes HPA로 CPU 기반 오토 스케일링.

③캐시 레이어 — Redis Cluster로 Hot Data 캐싱. 앱 레벨 로컬 캐시(Caffeine) 병행으로 Redis 부하도 분산.

④메시지 큐 — 실시간 응답이 불필요한 작업(알림, 정산 등)은 Kafka로 비동기 처리. 버스트 트래픽 버퍼링.

⑤데이터베이스 — Read Replica로 읽기 분산. 쓰기 집중 서비스는 DB 샤딩 + Cassandra 검토. CQRS로 읽기 모델 최적화.
캐시와 DB 사이의 데이터 불일치(정합성)를 어떻게 해결하나요?
Cache-Aside + TTL이 기본 전략입니다. 쓰기 시 DB 업데이트 후 캐시를 즉시 삭제(Evict)하고, 다음 읽기 때 DB에서 최신 데이터를 가져와 캐시에 채웁니다.

단순 삭제만으로는 Race Condition이 발생할 수 있습니다(삭제 후 이전 데이터로 재캐싱). 이 경우 Write-Through(쓰기 시 DB+캐시 동시 갱신) 또는 Versioned Cache Key(데이터 버전을 키에 포함)로 해결합니다.

TTL은 최후의 방어선으로 설정하되, 완전한 강한 일관성이 필요한 경우 해당 데이터는 캐싱하지 않는 것이 올바른 트레이드오프입니다.
MSA 환경에서 분산 트랜잭션을 어떻게 처리하나요?
2PC(Two-Phase Commit)는 가용성을 희생하므로 MSA에서는 잘 사용하지 않습니다.

일반적으로 SAGA 패턴을 사용합니다. 각 서비스가 로컬 트랜잭션을 실행하고 이벤트/메시지를 발행합니다. 실패 시 각 서비스가 보상 트랜잭션(Compensating Transaction)을 실행해 롤백합니다.

Choreography SAGA는 서비스 간 이벤트로 연쇄 처리(단순하지만 흐름 파악 어려움), Orchestration SAGA는 중앙 오케스트레이터가 순서 제어(복잡도↑ 대신 가시성↑). 결국 최종 일관성(Eventual Consistency)을 허용하는 설계가 핵심입니다.
DB 샤딩의 단점과 대안은 무엇인가요?
샤딩의 주요 단점: ① 크로스 샤드 JOIN/집계 불가 ② 샤드 키 변경이 매우 어려움 ③ 핫스팟 발생 가능 ④ 트랜잭션 복잡도 증가 ⑤ 운영 복잡도 폭발적 증가

대안 고려 순서:
1. 인덱스 최적화 + 쿼리 튜닝 — 가장 먼저 시도
2. Read Replica — 읽기 부하 분산
3. 데이터 아카이빙 — 오래된 데이터 분리
4. Vertical Scaling — 더 큰 서버
5. CQRS + 전용 읽기 저장소 (Elasticsearch 등)
6. 최후 수단으로 샤딩 또는 Cassandra 같은 분산 DB로 전환
서킷 브레이커 없이 하나의 서비스 장애가 전체 시스템에 미치는 영향을 설명하세요.
Cascading Failure(연쇄 장애) 시나리오입니다.

서비스 A가 응답이 느린 서비스 B를 호출한다고 가정합니다. B가 느려지면 A의 요청 스레드가 응답 대기로 묶입니다(Thread Pool 소진). A에 새 요청이 들어와도 스레드가 없어 큐에 쌓입니다. 결국 A도 응답 불가 상태가 되고, A를 호출하는 C, D도 동일하게 전파됩니다.

서킷 브레이커는 B 호출 실패율이 임계치를 넘으면 즉시 OPEN 상태로 전환해 B 호출 자체를 차단합니다. A의 스레드는 해방되고 Fallback 응답을 반환합니다. 이것이 장애 격리(Fault Isolation)의 핵심입니다.
배포 중 다운타임 없이 서비스를 유지하는 방법을 설명하세요.
핵심: Graceful Shutdown + 무중단 배포 전략

① Rolling Update — 기존 인스턴스를 하나씩 교체. 롤백 느림.
② Blue-Green — 동일한 두 환경 운영. 트래픽 전환 후 구버전 대기. 즉시 롤백 가능. 비용 2배.
③ Canary — 1~5% 트래픽만 신버전으로 점진 확대. 리스크 최소화.

앱 레벨 준비:
- SIGTERM 수신 시 신규 요청 차단, 처리 중인 요청 완료 후 종료
- K8s의 preStop Hook + terminationGracePeriodSeconds 설정
- 로드 밸런서가 Connection Draining으로 기존 연결 유지
