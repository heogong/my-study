마이크로서비스 아키텍처 · 핵심 패턴 · 기술 면접 대비
01 Monolith vs MSA Architecture
비교 개요
구분 모놀리식 마이크로서비스
배포 단위 하나의 단일 배포 단위 서비스별 독립 배포
기술 스택 단일 언어/프레임워크 폴리글랏 (서비스별 최적 기술 선택 가능)
스케일링 전체를 함께 확장 병목 서비스만 선택적 확장
장애 격리 한 곳 장애 → 전체 장애 서비스 단위 격리 (Circuit Breaker)
데이터 관리 단일 DB 공유 DB per Service (각 서비스 전용 DB)
개발 복잡도 낮음 (초기) 높음 (네트워크 통신, 분산 트랜잭션)
운영 복잡도 낮음 높음 (컨테이너 오케스트레이션, 모니터링)
팀 구조 기능별 팀 (프론트/백/DB) 도메인별 자율팀 (Conway's Law)
언제 MSA를 선택하는가?
팀 규모가 크고, 서비스별 스케일링 요구가 다르며, 빠른 독립 배포가 중요할 때. 소규모 팀이나 초기 스타트업에서는 모놀리스 퍼스트 전략이 더 현명하다.

API Gateway 역할

  • 인증/인가 — JWT 검증, OAuth2 처리를 중앙 집중화

  • Server-Side Discovery — 로드밸런서가 레지스트리 조회 후 라우팅 (AWS ELB + Route53)
  • Self Registration — 서비스 기동 시 레지스트리에 자신 등록
  • Health Check — 주기적으로 서비스 상태 확인, 비정상 인스턴스 제거

비동기 메시징

04 데이터 관리 패턴 Data Patterns
DB
DB per Service
서비스별 독립 데이터베이스
각 마이크로서비스는 전용 DB를 보유. 다른 서비스의 DB에 직접 접근 금지. 서비스 API를 통해서만 데이터 조회/변경 가능.
장점 서비스별 최적 DB 선택 가능 (폴리글랏 퍼시스턴스), 독립 스케일링, 장애 격리
단점 조인 불가 → API Composition 또는 CQRS로 해결, 분산 트랜잭션 복잡도 증가
TX
Saga Pattern
분산 트랜잭션
2PC 없이 분산 트랜잭션 관리. 로컬 트랜잭션 체인으로 구성하고, 실패 시 보상 트랜잭션(Compensating Transaction) 실행.
Choreography (코레오그래피) 각 서비스가 이벤트 발행/구독으로 자율 조정. 중앙 조정자 없음. 서비스 수 증가 시 흐름 추적 어려움
Orchestration (오케스트레이션) 중앙 Saga Orchestrator가 각 서비스에 명령. 흐름 파악 쉬움. 단일 실패 지점 존재
QUERY
CQRS
명령/조회 책임 분리
Command(쓰기)와 Query(읽기)를 분리. 쓰기 모델은 도메인 규칙 중심, 읽기 모델은 조회 최적화된 별도 View DB 유지.
적용 시나리오 읽기/쓰기 비율 불균형 시스템, 복잡한 집계 쿼리 필요 시, 이벤트 소싱과 함께 사용
주의: 최종 일관성(Eventual Consistency). 쓰기 후 즉시 읽기 시 오래된 데이터 반환 가능
EVENT
Event Sourcing
이벤트 기반 상태 저장
현재 상태 대신 이벤트의 시퀀스를 저장. 현재 상태는 이벤트를 재생(Replay)하여 계산. 완전한 변경 이력 보존.
장점 완벽한 감사 로그, 시간 여행(Time Travel), CQRS와 궁합 좋음
단점 이벤트 스키마 변경 어려움, 스냅샷 없으면 Replay 성능 저하
SYNC
Outbox Pattern
트랜잭션 아웃박스
DB 저장과 이벤트 발행의 원자성 보장. 이벤트를 메시지 브로커 직접 발행 대신 DB의 Outbox 테이블에 함께 저장하고, 별도 프로세스가 발행.
구현 방식 Debezium (CDC) 또는 Polling Publisher로 Outbox 테이블 감지 후 Kafka 발행
READ
API Composition
API 조합
여러 서비스의 데이터를 API 호출로 조합하여 반환. DB 조인 대신 여러 서비스에 쿼리 후 메모리에서 조인. BFF 패턴과 함께 사용.
단점 N+1 문제 주의. 여러 서비스 중 하나 장애 시 전체 응답 실패 가능 → Circuit Breaker 필수
05 장애 처리 패턴 (Resilience) Fault Tolerance
Circuit Breaker
  • 장애 서비스 호출을 빠르게 차단하여 연쇄 장애(Cascade Failure) 방지
  • Closed — 정상 상태, 모든 요청 통과
  • Open — 임계치 초과 시, 모든 요청 즉시 실패 반환 (Fallback)
  • Half-Open — 일정 시간 후 소량 요청 테스트, 성공 시 Closed로 전환
Resilience4j Hystrix (deprecated) Spring Cloud
// Resilience4j Circuit Breaker 예시 CircuitBreaker cb = CircuitBreaker.ofDefaults("orderService"); Supplier<Order> decorated = CircuitBreaker .decorateSupplier(cb, orderService::getOrder); Try.ofSupplier(decorated) .recover(ex -> fallbackOrder());
기타 탄력성 패턴
  • Retry — 일시적 장애 시 재시도. Exponential Backoff + Jitter로 재시도 폭풍 방지
  • Timeout — 응답 대기 시간 제한. 느린 서비스로 인한 스레드 고갈 방지
  • Bulkhead — 서비스별 스레드 풀 격리. 한 서비스 장애가 다른 서비스 스레드 고갈 방지 (선박 격벽 구조)
  • Rate Limiter — 초당 요청 수 제한. 과부하 방지
  • Fallback — 장애 시 기본값/캐시값 반환. 서비스 저하(Degradation) 허용
  • Health Check — /health 엔드포인트로 정상 여부 확인. K8s liveness/readiness probe
연쇄 장애(Cascade Failure) 시나리오: Service A → Service B → Service C 체인에서 C 장애 시, B의 스레드가 모두 C 응답 대기로 고갈되고, A도 같은 방식으로 고갈되어 전체 시스템 다운. Circuit Breaker + Timeout + Bulkhead의 조합으로 방어.
06 배포 전략 Deployment
Blue/Green Deployment
  • 현재 버전(Blue)과 새 버전(Green) 환경을 동시 유지
  • 트래픽을 한 번에 Green으로 전환, 문제 시 즉시 Blue로 롤백
  • 장점 — 무중단, 즉시 롤백, 운영 환경 그대로 테스트
  • 단점 — 인프라 비용 2배, DB 스키마 변경 주의
Canary Deployment
  • 새 버전을 소수 인스턴스에만 배포 (5~10%), 점진적 트래픽 증가
  • 실제 사용자 트래픽으로 검증 후 전체 롤아웃
  • 장점 — 실제 환경 검증, 위험 최소화
  • 단점 — 두 버전 동시 운영 복잡도, 모니터링 필수
Rolling Update
  • 인스턴스를 순차적으로 교체 (배포 중 구버전/신버전 동시 운영)
  • K8s 기본 배포 전략. maxSurge / maxUnavailable 설정
  • 장점 — 별도 인프라 불필요, 점진적 교체
  • 단점 — 버전 혼재 기간 존재, API 하위 호환 필수
컨테이너 & 오케스트레이션
  • Docker — 서비스별 컨테이너 이미지 빌드, 환경 일관성 보장
  • Kubernetes — 컨테이너 오케스트레이션, 자동 스케일링(HPA), 자가 치유
  • Helm Chart — K8s 배포 패키지 관리
  • CI/CD — Jenkins/GitHub Actions → 이미지 빌드 → 레지스트리 → K8s 배포

08 MSA 보안 Security
인증/인가 (JWT + OAuth2)
  • JWT — 서명된 토큰으로 사용자 정보 담아 전달. API Gateway에서 검증, 하위 서비스는 신뢰
  • OAuth2 / OIDC — 서드파티 인증. Authorization Code Flow 권장
  • 중앙 Auth Service — 토큰 발급/검증/갱신 전담 서비스
  • API Key — 서비스 간 내부 통신에 API Key 또는 mTLS 사용
// JWT Payload 구조 { "sub": "user-123", "roles": ["ROLE_USER"], "exp": 1720000000, "iss": "auth-service" }
서비스 간 보안
  • mTLS — 서비스 간 상호 TLS 인증. 인증서 기반으로 서비스 신원 확인. Istio로 자동 관리
  • 네트워크 정책 — K8s NetworkPolicy로 허용된 서비스만 통신
  • Secret 관리 — 환경변수에 직접 저장 금지. HashiCorp Vault, AWS Secrets Manager
  • Zero Trust — 내부 네트워크도 신뢰하지 않음. 모든 요청 검증
  • RBAC — Role 기반 접근 제어. 서비스별 최소 권한 원칙
09 서비스 분리 원칙 (DDD) Domain Design
Bounded Context
  • 경계 컨텍스트 — 도메인 모델이 명확히 정의되는 경계. 각 BC 내에서는 용어/모델이 일관성 유지
  • "Order"가 주문 서비스에서는 배송 정보 포함, 재고 서비스에서는 SKU/수량으로 다르게 정의
  • BC가 MSA 서비스 분리 단위의 자연스러운 경계
  • Context Map — BC 간 관계 다이어그램. 통합 방식 명시
서비스 분리 기준
  • 단일 책임 — 변경 이유가 하나뿐인 서비스 단위로 분리
  • 팀 소유권 — Conway's Law: 팀 구조가 아키텍처를 따라감. 팀 단위로 서비스 소유
  • 독립 배포 — 다른 서비스 수정 없이 독립 배포 가능해야 함
  • 데이터 경계 — 데이터를 공유하는 서비스는 분리하지 말 것
  • 너무 작은 분리 주의 — Nano Service Anti-pattern: 네트워크 홉만 증가
Strangler Fig Pattern: 모놀리스에서 MSA로 점진적 전환 전략. 새 기능은 새 서비스로, 기존 기능은 점진적으로 마이그레이션. API Gateway가 구/신 버전 라우팅. 한번에 전환하는 Big Bang 방식보다 안전.
10 기술 면접 핵심 질문 Q&A
Q01MSA에서 분산 트랜잭션을 어떻게 처리하나요?
MSA에서는 각 서비스가 독립 DB를 가지므로 RDBMS의 2PC(Two-Phase Commit)를 사용할 수 없습니다. 대신 Saga 패턴을 사용합니다.

Choreography Saga — 각 서비스가 로컬 트랜잭션 완료 후 이벤트 발행, 다음 서비스가 구독하여 처리. 느슨한 결합이지만 흐름 파악이 어렵습니다.
Orchestration Saga — 중앙 Saga Orchestrator가 각 서비스에 명령 전달. 흐름이 명확하지만 단일 실패 지점이 생깁니다.

실패 시에는 보상 트랜잭션(Compensating Transaction)으로 이미 수행한 작업을 롤백합니다. 결과적 일관성(Eventual Consistency)을 허용하는 설계가 전제됩니다.
Q02Circuit Breaker의 동작 원리와 상태 전이를 설명해주세요.
Circuit Breaker는 외부 서비스 장애 시 연쇄 실패를 막는 패턴입니다. 세 가지 상태로 동작합니다.

Closed (정상) — 모든 요청을 통과시키며 실패율을 모니터링합니다.
Open (차단) — 설정된 실패율 임계치(예: 50%) 초과 시 전환. 이후 모든 요청을 즉시 실패 처리하고 Fallback 응답을 반환합니다. 다운스트림 서비스 부하를 줄이고 시스템이 복구할 시간을 줍니다.
Half-Open (테스트) — Open 상태에서 일정 시간 후 소수의 요청을 허용. 성공하면 Closed로, 실패하면 Open으로 돌아갑니다.

Spring Boot에서는 Resilience4j를 주로 사용합니다.
Q03API Gateway와 Service Mesh의 차이점은 무엇인가요?
두 가지 모두 네트워크 제어 계층이지만 동작하는 위치와 목적이 다릅니다.

API Gateway — 외부(North-South) 트래픽 처리. 클라이언트와 내부 서비스 사이의 단일 진입점. 인증/인가, 라우팅, Rate Limiting, SSL 종단을 담당합니다.
Service Mesh — 내부(East-West) 트래픽 처리. 서비스 간 통신을 제어. 사이드카 프록시(Envoy)가 각 서비스 Pod에 주입되어 트래픽 암호화(mTLS), Circuit Breaker, 분산 추적을 제공합니다.

실제로는 두 가지를 함께 사용합니다. API Gateway(Kong/AWS API GW)가 외부 요청을 받고, 내부에서는 Istio 같은 Service Mesh가 서비스 간 통신을 관리합니다.
Q04MSA에서 데이터 일관성을 어떻게 보장하나요?
MSA는 강한 일관성(Strong Consistency) 대신 최종 일관성(Eventual Consistency)을 허용하는 것이 핵심입니다.

Outbox Pattern — DB 저장과 이벤트 발행을 하나의 로컬 트랜잭션으로 묶어 원자성 보장. Outbox 테이블에 이벤트 저장 후 CDC(Debezium)가 Kafka로 발행합니다.
Idempotency(멱등성) — 이벤트가 중복 전달되어도 같은 결과가 나오도록 설계. 이벤트 ID로 중복 처리 방지.
CQRS + Event Sourcing — 이벤트를 원천으로 읽기 모델을 독립적으로 구성.

설계 시 "데이터가 얼마나 오래된 것을 허용할 수 있는가"를 먼저 결정하고, 비즈니스 요구사항에 따라 일관성 수준을 조정합니다.
Q05서비스 분리 기준은 무엇인가요? 너무 잘게 쪼개면 문제가 생기나요?
DDD의 Bounded Context가 가장 좋은 분리 단위입니다. 단일 책임, 독립 배포 가능성, 팀 소유권을 기준으로 판단합니다.

너무 잘게 분리하면(Nano Service Anti-pattern) — 네트워크 홉 증가로 지연(Latency) 누적, 운영 복잡도 폭증, 트랜잭션 경계가 과도하게 분산, 팀 오버헤드 증가 문제가 생깁니다.

반면 너무 크게 유지하면(Distributed Monolith) — 서비스 간 강한 결합으로 독립 배포의 장점을 잃습니다.

이상적인 크기는 "두 피자 팀(2 Pizza Team)"이 소유할 수 있는 서비스. 실제 데이터 변경 패턴과 팀 구조를 함께 고려해 결정합니다.
Q06MSA 환경에서 로그를 어떻게 추적하나요?
여러 서비스에 분산된 로그를 추적하려면 Correlation ID(Trace ID)를 모든 요청에 전파해야 합니다.

구조화 로그(JSON) — 모든 로그를 JSON 형태로 출력해 파싱 용이하게 합니다.
MDC(Mapped Diagnostic Context) — Spring + Logback에서 Trace ID를 스레드 로컬에 저장하여 모든 로그에 자동 포함.
중앙 집중 로그 — Filebeat → Logstash → Elasticsearch → Kibana (ELK) 스택으로 수집, 검색, 시각화.
분산 추적 — Spring Cloud Sleuth가 자동으로 Trace ID/Span ID를 주입하고 Zipkin에 전송. 서비스 간 호출 체인을 시각화합니다.
Q07모놀리스에서 MSA로 전환할 때 전략은?
Strangler Fig Pattern이 가장 현실적인 전략입니다. 모놀리스를 한 번에 전환하는 Big Bang 방식은 위험이 높습니다.

1단계 — API Gateway(또는 Nginx)를 모놀리스 앞에 배치. 모든 트래픽이 Gateway를 통과하도록 변경.
2단계 — 신규 기능을 별도 서비스로 개발하고 Gateway에서 라우팅.
3단계 — 기존 기능을 점진적으로 서비스로 추출. 데이터 마이그레이션 포함.
4단계 — 모놀리스 비중이 줄어들면 최종 제거.

DB 분리가 가장 어렵습니다. 공유 DB 단계 → 별도 스키마 단계 → 별도 DB 단계로 점진적으로 분리합니다.
